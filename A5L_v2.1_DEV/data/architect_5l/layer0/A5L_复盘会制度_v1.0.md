## 📅 三级复盘架构

```
┌─────────────────────────────────────────────────────────────┐
│  日复盘 (Daily Review)                                       │
│  ├── 时间: 每天 17:30-17:45 (15分钟)                          │
│  ├── 参与者: L0六管理者 + Chief Architect                     │
│  ├── 目的: 发现问题、快速调整、记录教训                        │
│  └── 产出: 日复盘记录 + 问题清单                              │
├─────────────────────────────────────────────────────────────┤
│  周复盘 (Weekly Review)                                      │
│  ├── 时间: 每周日 21:00-21:30 (30分钟)                        │
│  ├── 参与者: L0六管理者 + 各Layer负责人                       │
│  ├── 目的: 周度总结、趋势分析、策略调整                        │
│  └── 产出: 周复盘报告 + 下周计划                              │
├─────────────────────────────────────────────────────────────┤
│  月复盘 (Monthly Review)                                     │
│  ├── 时间: 每月最后一天 20:00-21:00 (60分钟)                  │
│  ├── 参与者: 全员 (L0 + 各Layer)                             │
│  ├── 目的: 深度复盘、战略调整、能力建设                        │
│  └── 产出: 月复盘报告 + 战略调整方案 + 经验教训库              │
└─────────────────────────────────────────────────────────────┘
```

## 📋 日复盘会流程 (17:30-17:45)

### 1. 数据回顾 (3分钟)
- 今日Git提交数
- 系统健康度变化
- 完成的任务清单

### 2. 问题曝光 (7分钟)
每个管理者用 **1分钟** 回答：
1. 今天发现了什么问题？
2. 为什么发生？
3. 如何立即解决？

### 3. 教训沉淀 (3分钟)
- 新问题 → 记录到**问题库**
- 重复问题 → 升级处理
- 成功经验 → 记录到**最佳实践**

### 4. 明日调整 (2分钟)
- Chief Architect确认调整措施
- COO分配明日任务

## 🎭 L0管理者复盘职责

| 角色 | 复盘重点 | 必答问题 |
|------|----------|----------|
| **Chief Architect** | 系统架构问题、决策失误 | "今天的架构决策有问题吗？" |
| **Chief Investment Officer** | 投资决策、机会识别 | "错过了什么机会？判断错了什么？" |
| **Chief Operating Officer** | 执行效率、资源协调 | "哪里执行卡壳了？资源够吗？" |
| **Chief Security Officer** | 风险遗漏、安全隐患 | "漏掉了什么风险？合规有问题吗？" |
| **Knowledge Guardian** | 知识管理、归档质量 | "知识归档完整吗？好检索吗？" |
| **Report Manager** | 分析质量、产出效率 | "分析报告质量够吗？产出及时吗？" |

## 🚫 复盘红线

1. **不追责** - 复盘是为了改进，不是找替罪羊
2. **不遗漏** - 所有问题必须记录，小问题也要提
3. **不拖延** - 发现的问题24小时内要有措施
4. **不重复** - 重复出现的问题必须系统解决
5. **不形式** - 复盘必须有产出，不能走过场

## 📚 经验教训库

### 分类体系
```
lessons_learned/
├── 决策失误/           # 判断错误、决策失误
├── 流程漏洞/           # 流程缺失、执行偏差
├── 协作问题/           # 沟通不畅、协作失效
├── 技术债务/           # 架构问题、技术短板
├── 外部风险/           # 市场变化、政策影响
└── 最佳实践/           # 成功经验、方法论
```

### 经验教训格式
```yaml
id: LESSON-2026-0504-001
date: 2026-05-04
category: 流程漏洞
title: Chief包办代替，未调用KG
severity: 高
frequency: 首次
description: |
  收到韩文研报图片后，Chief Architect直接尝试分析，
  没有调用Knowledge Guardian执行专业分析。
root_cause: |
  1. 角色认知偏差：Chief混淆了"协调者"和"执行者"
  2. 流程缺失：没有标准化的"任务分配→执行→审核"流程
  3. 惯性思维：习惯性地想要快速响应，忽略了协作
impact: |
  - 效率降低：Chief不擅长OCR和韩文分析
  - 质量风险：可能遗漏专业细节
  - 能力浪费：KG的能力没有被利用
lesson: |
  Chief Architect的职责是"识别任务类型→分配最合适的L0管理者→审核结果"，
  而不是"识别任务→自己执行"。
action: |
  1. 建立任务分类清单，明确每种任务的执行者
  2. Chief收到任务后，先判断"这是谁的专业领域"
  3. 强制流程：重要任务必须走"分配→执行→审核"
prevented: false
status: 已记录，待验证
```

## 📊 复盘效果度量

### 关键指标
| 指标 | 目标 | 度量方式 |
|------|------|----------|
| 日复盘出席率 | 100% | 6位管理者全部提交日报 |
| 问题记录率 | 100% | 所有问题都记录在案 |
| 重复问题率 | <10% | 同一问题不重复出现 |
| 措施完成率 | >90% | 复盘确定的措施按时完成 |
| 经验教训数 | 每周≥3条 | 形成组织知识沉淀 |

### 复盘质量检查
- 是否所有管理者都发言了？
- 是否发现了真正的问题？
- 是否有明确的解决措施？
- 是否记录了经验教训？
- 是否跟进了之前的问题？
