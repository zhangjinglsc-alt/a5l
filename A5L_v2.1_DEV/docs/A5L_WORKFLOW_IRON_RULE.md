## 一、数据真实性铁律（绝对红线）

### 1.1 零容忍原则

```
┌─────────────────────────────────────────────────────────────┐
│                     数据真实性铁律                          │
├─────────────────────────────────────────────────────────────┤
│  ❌ 禁止行为（一旦发现，立即问责）：                         │
│  1. 编造不存在的数据                                        │
│  2. 用模拟数据冒充真实数据                                    │
│  3. 隐瞒数据来源问题                                         │
│  4. 在数据异常时正常发送报告                                  │
│  5. 未经验证就使用新数据源                                    │
│                                                             │
│  ✅ 强制行为：                                               │
│  1. 所有数据必须标注来源和校验状态                            │
│  2. 数据异常时立即告警，绝不隐瞒                              │
│  3. 校验失败时宁可不发报告                                    │
│  4. 主动承认数据获取失败，不伪装成功                          │
│  5. 用缓存数据时必须明确标注"缓存数据，可能延迟"                │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 数据分级制度

| 等级 | 说明 | 处理方式 |
|------|------|----------|
| **A级-已验证** | 多源交叉验证通过，置信度>95% | 正常使用 |
| **B级-待验证** | 单源获取或交叉验证差异<5% | 标注"待验证"，降低置信度 |
| **C级-存疑** | 交叉验证差异5-10%，或数据源不稳定 | 标注"数据存疑"，谨慎使用 |
| **D级-不可用** | 交叉验证差异>10%，或所有源故障 | **禁止使用，立即告警** |

**规则**: 日报中只能使用A级和B级数据，C级需Chief确认，D级坚决不用。

## 二、完整工作流（含兜底方案）

### 2.1 日报生成工作流

```
【17:00】启动准备
├── Data Engineer: 检查所有API可用性
│   ├── A股数据源（AKShare/TuShare）
│   ├── 港股数据源（AKShare/新浪财经）
│   └── 美股数据源（Yahoo/Alpha Vantage）
│   └── 结果: 所有源正常→继续 / 有故障→启动故障流程
│
【17:05】数据采集
├── Data Engineer: 多源获取数据
│   ├── 同一指标至少2个独立源
│   └── 记录每个数据点的来源
│
【17:10】交叉验证
├── CIO: 执行交叉验证
│   ├── 对比多源数据，标记差异
│   ├── 计算数据质量评分
│   └── 结果: 评分≥70→继续 / <70→启动数据异常流程
│
【17:15】等待截图
├── CSO: 等待Chief发送截图
│   ├── 超时10分钟→告警Chief
│   └── 收到截图→立即OCR识别
│
【17:20】持仓数据校验
├── CSO: 执行OCR校验
│   ├── 截图 vs 系统记录比对
│   ├── 担保比例独立计算
│   └── 结果: 一致→继续 / 不一致→以截图为准，告警
│
【17:25】SKILL调用
├── A5L L3: 调用37个SKILL
│   ├── 执行分析
│   └── 多SKILL一致性检查
│
【17:28】最终校验
├── Report Manager: 最终输出校验
│   ├── 完整性检查
│   ├── 数据溯源检查
│   └── 质量评分检查
│
【17:30】发送或告警
├── 质量评分≥70: 发送日报
├── 质量评分<70: 发送告警+延迟说明
└── 关键数据D级: 发送数据故障通知，不发日报
```

### 2.2 故障处理流程

#### 场景1: 单一API故障
```
时间: 17:00-17:05
责任人: Data Engineer
动作:
  1. 5分钟内自动切换备用源
  2. 记录故障日志
  3. 继续正常流程
兜底: 备用源也故障→升级至场景3
```

#### 场景2: 单一数据源不稳定（数据质量C级）
```
时间: 17:10
责任人: CIO
动作:
  1. 标记该指标为"数据存疑"
  2. 降低该部分分析的置信度
  3. 报告中明确标注数据来源问题
  4. 继续生成日报，但Chief知情
兜底: Chief可要求跳过该指标的分析
```

#### 场景3: 所有行情API故障（数据质量D级）
```
时间: 17:00-17:15
责任人: Data Engineer→CA
动作:
  1. Data Engineer 15分钟内尝试所有备用源
  2. 全部失败→立即告警CA
  3. CA决策:
     ├── 选项A: 使用昨日收盘数据+标注"无实时数据"
     ├── 选项B: 延迟至数据恢复后发送
     └── 选项C: 仅发送持仓分析，跳过市场分析
  4. 绝不编造数据！
兜底: CA明确告知Chief数据故障，由Chief决策
```

#### 场景4: OCR识别失败
```
时间: 17:15-17:25
责任人: CSO
动作:
  1. 立即请求Chief重新发送截图
  2. 30分钟内重试最多3次
  3. 仍失败:
     ├── 选项A: 跳过该账户的分析
     ├── 选项B: 使用系统数据+标注"未校验"
     └── 绝不编造持仓数据！
兜底: 明确告知Chief该账户数据不可用
```

#### 场景5: 担保比例计算异常
```
时间: 任何时间
责任人: CSO→CA
动作:
  1. 立即暂停该账户的杠杆建议
  2. 告警CA和Chief
  3. 人工确认正确数据
  4. 修正后才恢复分析
兜底: 宁可不分析，绝不给错误的杠杆建议
```

## 四、数据真实性保障措施

### 4.1 技术手段

1. **多源冗余**
   - 每个关键指标至少3个独立数据源
   - 自动切换，无需人工干预

2. **实时校验**
   - API获取时立即校验有效性
   - 交叉验证差异自动计算

3. **溯源追踪**
   - 每个数据点记录来源和时间戳
   - 可追溯，可审计

4. **缓存机制**
   - 本地缓存最近3日数据
   - API故障时明确标注"缓存数据"

### 4.2 制度保障

1. **分级审批**
   - A/B级数据: 自动生成
   - C级数据: Chief确认后使用
   - D级数据: **坚决不用**

2. **双重确认**
   - 关键数据（担保比例/持仓数量）必须双重验证
   - 系统计算 + 独立计算比对

3. **延迟机制**
   - 数据异常时宁可延迟发送，绝不仓促出错
   - 延迟必须明确告知Chief原因

4. **事后审计**
   - 每周审计数据质量日志
   - 发现问题立即整改

## 五、责任与问责

### 5.1 责任矩阵

| 角色 | 责任 | 失职后果 |
|------|------|----------|
| Data Engineer | API数据准确性 | 假数据流入→降级 |
| CIO | 交叉验证完整性 | 未验证数据使用→降级 |
| CSO | 持仓数据真实性 | 截图数据错误未检出→严重问责 |
| A5L L3 | 逻辑一致性 | 矛盾数据未标记→警告 |
| Report Manager | 最终输出完整性 | 发送问题报告→降级 |
| **CA** | **整体架构和兜底** | **系统性失误→Chief问责** |

### 5.2 问责机制

**轻微失职**（偶发小错误）:
- 警告记录
- 限期整改
- 不影响继续使用

**严重失职**（假数据/隐瞒问题）:
- 立即停用相关模块
- 书面检讨
- Chief审批后才能恢复

**系统性失职**（多次/重大错误）:
- **CA承担责任**
- 架构重新设计
- 流程重新审批
